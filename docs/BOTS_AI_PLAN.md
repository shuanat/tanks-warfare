# План имплементации ботов с ИИ

Документ фиксирует пошаговый план внедрения ботов с опорой на текущую архитектуру Tanks: сервер в основном ретранслирует состояние и события, физика и ввод — на клиенте. См. также [`protocol.md`](./protocol.md), [`REFACTORING_PLAN.md`](./REFACTORING_PLAN.md), [`ARCHITECTURE.md`](./ARCHITECTURE.md).

---

## Цель и ограничения

**Цель:** игроки-боты в лобби и в матче, разумное поведение, без поломки протокола.

**Ограничение архитектуры:** полноценной серверной симуляции танка сейчас нет; `handleState` сохраняет `lastPos` и рассылает `state` остальным (`server/ws/handlers/gameState.ts`). Значит, «ИИ» должен либо **генерировать те же сообщения, что живой клиент**, либо развиваться в сторону **общей/серверной физики** (дольше, но правильнее для честности).

---

## Рекомендуемая стратегия (два пути)

### Путь A — быстрый MVP (эмуляция клиента на сервере)

- Бот — запись в составе лобби **без** реального `WebSocket`, но с теми же полями, что ожидают хендлеры (`id`, `team`, `color`, `spawnTime`, …), плюс флаг `isBot`.
- Раз в тик (например 10–20 Гц или таймер на матч) «мозг» бота вызывает ту же логику, что и входящие `ClientMsg.STATE`, `BULLET` и т.д. — не через фейковый WS, а через **внутренний API**: рефактор `handle*` так, чтобы ядро было функцией вида `(lobby, actor, data)`.

**Плюсы:** мало новых систем, быстро играбельно.  
**Минусы:** дублирование движения с клиентом или риск рассинхрона, если логика разъедется.

### Путь B — устойчивый (общая симуляция шага)

- Вынести **детерминированный** шаг движения и клампа к карте (и при необходимости простые проверки выстрела) в `shared/` или чистый модуль, который используют **и клиент, и бот**.
- Бот на сервере решает только **вектор ввода**; позиция считается тем же кодом.

**Плюсы:** один источник правды, проще юнит-тесты (см. `REFACTORING_PLAN.md`).  
**Минусы:** заметный объём работ до первого «бот едет как игрок».

**Практично:** начать с A для поведения и UI, параллельно планировать B для движения.

## Фазы имплементации

### Фаза 0 — Транспортный контракт участника матча

Цель: убрать жёсткую зависимость игровой логики от `WebSocket` до добавления ботов.

- Ввести общий тип участника матча (например `GameActor` / `LobbyMember`) с минимальным контрактом:
  - `id`, `team`, `nickname`, `color`, `isBot`, `isInGame`, `spawnTime`;
  - метод доставки сообщений (например `emit(event)`), чтобы логика боя не вызывала `ws.send` напрямую.
- Для реального игрока сделать адаптер поверх `WebSocket`; для бота — адаптер без сетевого транспорта.
- Все рассылки (`lobby_state`, `start`, события боя) перевести на этот контракт.

**Критерий:** в `broadcast` и `handlers` нет прямой необходимости знать, что участник — именно `WebSocket`.

### Фаза 1 — Модель «игрок-бот» в лобби

- Расширить представление игрока в лобби (сейчас в `lobbyStore.ts` — массив `WebSocket[]`): ввести **discriminated union** или общий интерфейс `LobbyMember` с полями `id`, `team`, `nickname`, `isBot`, опционально `difficulty`.
- В `handlers/lobby.ts`: создание бота хостом (команда / кнопка), удаление при выходе, учёт в `lobby_state` и в `start` (`allPlayers`).
- Клиент: отображение ника (например `[AI]`), отсутствие ожидания «живого» клиента для слота бота.

**Критерий:** бот в составе лобби, старт матча стабилен, все клиенты видят слот и тип участника (`isBot`).

### Фаза 2 — Серверный тик боя и сенсоры

- Подписка на «матч идёт»: один `setInterval` / таймер на лобби.
- Явно определить владение таймером: `lobby.aiTickHandle` + функции `startAiTick(lobby)` / `stopAiTick(lobby)`.
- Обязательные точки остановки: `game_over`, удаление лобби, матч завершён вручную, серверный `close` последнего человека, uncaught error в тике.
- **Мир для ИИ (read model):** агрегировать данные сервера: `lastPos` по игрокам, `lobby.mines`, `lobby.boosts`, `lobby.rockets`, снимок пуль из `handlers/combat.ts` (при необходимости добавить лёгкую структуру для ботов).
- Для `lastPos` хранить `updatedAt` и TTL (например 300–500 мс). Просроченные данные не использовать для стрельбы и погони.
- Зафиксировать в комментарии или в этом документе: для каждого действия бота — **сервер: авторитетно / клиент: предсказание** (как в `REFACTORING_PLAN.md`).

### Фаза 3 — Поведение (слои ИИ)

1. **Базовый конечный автомат:** патруль → при обнаружении врага (дистанция + при появлении — луч и препятствия как в системе обнаружения из `ROADMAP_AND_ISSUES.md`) → прицеливание → выстрел с кулдауном → отход при низком HP.
2. **Навигация:** сетка по `mapData.bricks` (A\* или упрощённый обход) — отдельный чистый модуль + юнит-тесты на карте-фикстуре.
3. **Сложность:** параметр `difficulty` — точность прицела, задержка реакции, агрессия.

На старте достаточно эвристик и тестов; отдельный ML/RL — опционально позже.

### Фаза 4 — Протокол (минимально)

- Если хосту нужны команды «добавить/убрать бота» — добавить `ClientMsg` / `ServerMsg` в `shared/src/protocol.ts` и `shared/src/payloads.ts`, хендлер в `dispatch.ts`, обработку на клиенте в `messageHandlers.ts`.
- Альтернатива на первом шаге: отладочная/серверная команда без UI, без расширения протокола.

Минимальные правила валидации команд управления ботами:

| Команда      | Кто может вызывать | Когда разрешено                         | Валидация                                               | Ошибка                                                             |
| ------------ | ------------------ | --------------------------------------- | ------------------------------------------------------- | ------------------------------------------------------------------ |
| `add_bot`    | Только хост лобби  | Только до `gameStarted`                 | `bots + humans <= MAX_PLAYERS`, допустимый `difficulty` | `error: not_host / game_started / lobby_full / invalid_difficulty` |
| `remove_bot` | Только хост лобби  | До и во время матча (по решению режима) | бот с `botId` существует в этом лобби                   | `error: not_host / bot_not_found`                                  |

Если удаление во время матча запрещено, зафиксировать это явно и отразить в UI.

### Фаза 5 — Качество и честность

- Бот не должен слать невозможные `state` (скорость, телепорты): clamp по константам из `shared` / `constants`.
- Когда в игре появится **система обнаружения** (роадмап), ИИ должен использовать **те же правила видимости**, что и игроки.

Измеримые критерии готовности (DoD) для первой версии:

- **Стабильность:** 15 минут матча с 4–8 ботами без неотчищенных таймеров и без роста числа активных `setInterval`.
- **Производительность:** серверный тик ИИ укладывается в целевой бюджет (например, `p95 < 4 ms` при 8 ботах).
- **Корректность:** нет `ws_unhandled_client_message_type` для новых команд ботов, нет падений `dispatch` на некорректных payload.
- **Играбельность:** бот выполняет базовый цикл (движение → прицеливание → выстрел → смена цели) и не «зависает» более N секунд.

### Фаза 6 (опционально) — внешний процесс с WebSocket

- Отдельный клиент-бот для экспериментов; для продукта предпочтительнее встроенный серверный контроллер.

## Риски и митигация

| Риск                                | Митигация                                                                              |
| ----------------------------------- | -------------------------------------------------------------------------------------- |
| Рассинхрон движения бота и клиентов | Общий шаг движения (путь B) или жёсткая синхронизация с клиентскими константами        |
| Хендлеры завязаны на `WebSocket`    | Выделить `GameActor`; для бота — объект с теми же полями без реальной отправки в сокет |
| Нагрузка CPU при многих ботах       | Редкий тик ИИ (5–10 Гц), упрощённая навигация, лимит ботов в лобби                     |
| Утечки таймеров и «зомби»-боты      | Явный lifecycle `startAiTick/stopAiTick`, очистка при всех сценариях завершения матча  |
| Устаревшие данные сенсоров          | TTL на `lastPos`, фильтрация stale-данных перед принятием решений                      |

## Краткий чеклист

1. Транспортный контракт участника матча (`GameActor`), отвязка от прямого `WebSocket`.
2. Абстракция участника лобби + флаг бота + рассылка в `start` / `lobby_state`.
3. Тик матча + lifecycle таймера + сбор сенсоров на сервере (включая TTL для `lastPos`).
4. Эвристический ИИ + вызов существующих путей рассылки (`broadcastGame` и т.д.).
5. Команды управления ботами и их валидация в протоколе (или зафиксированная серверная альтернатива).
6. Тесты и DoD-метрики: навигация, луч/препятствия, регрессия `dispatch`, профиль тика.
7. Усложнение под fog-of-war, когда фича будет в коде.
