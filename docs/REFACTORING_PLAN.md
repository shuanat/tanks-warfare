Документ фиксирует **технический** план улучшения кодовой базы. Игровой контент и фичи из `ROADMAP_AND_ISSUES.md` можно встраивать параллельно; здесь — **структура, качество, сопровождение**.

Связанные файлы: [ARCHITECTURE.md](./ARCHITECTURE.md), [../ROADMAP_AND_ISSUES.md](../ROADMAP_AND_ISSUES.md).

- **Сервер:** TypeScript в `server/**/*.ts` → сборка `npm run build:server` → `server/dist/`; в dev — `tsx watch server/index.ts`. Общий код — импорт `#shared/*` из корневого `package.json`. Статика — `STATIC_ROOT`.

## Юнит-тесты и рефакторинг (сквозная линия)

**Цель не «100% покрытие игры»:** весь `gameClient.js`, цикл отрисовки, WebSocket и canvas в юнитах воспроизводить дорого и хрупко. Для безопасного рефакторинга достаточно **таргетных тестов** на вынесенную **чистую** логику.

| Делать юнитами                                                                                                                                     | Не ставить целью на старте                         |
| -------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------- |
| Валидация и строки (`sanitizeNick`, имена лобби и т.д.)                                                                                            | «Покрыть весь бой» одним слоем тестов              |
| Генерация карты (`generateMapData`) — структура, границы, инварианты                                                                               | Каждый кадр `requestAnimationFrame`                |
| Коллизии, кламп карты, геометрия — **`game/collision.js`** + `collision.test.js` (фаза 3.2)                                                        | Пиксель-перфект совпадение с canvas без скриншотов |
| Разбор маршрутизации: например, что по `type` вызывается нужный хендлер (с моками) — после 2.1 на клиенте / уже возможно на сервере для `dispatch` | Полная симуляция матча в Node без моков WS         |

**Правило на merge:** вынесли модуль с **детерминированной** логикой — по возможности **+тесты в том же PR** или следующим маленьким; ручной сценарий «лобби → бой → …» остаётся обязательным, тесты его **дополняют**, не заменяют.

**Инструмент:** Vitest в корне; тесты рядом с кодом: **`client/src/**/*.test.js`**, **`server/**/*.test.ts`**. Для серверных импортов из `#shared/*` в тестах см. alias в `vitest.config.js`. Запуск в CI — фаза 5.1.

## Модель доверия: клиент и сервер

Сейчас часть логики **предсказывается на клиенте** и подтверждается/рассылается сервером. При усложнении игры (новые режимы, античит, обнаружение) важно явно решать, **что сервер обязан проверять** (позиции, урон, подбор буста, разрушение кирпича и т.д.), а что остаётся «косметикой» на клиенте.

Рекомендация для плана: при добавлении фич из роадмапа фиксировать в коммите одну строку — _«сервер: авторитетно / клиент: предсказание»_ для затронутого действия. Полный перенос авторитета на сервер — отдельная большая задача, не смешивать её с косметическим рефакторингом без необходимости.

## Приоритизация (рекомендуемый порядок работ)

1. **Фаза 0.5** — линт + CI; параллельно лёгкие серверные логи из фазы 1.
2. **Фаза 2.1** — вынести `handleData`; отдельный merge, без обязательного GameState.
3. **Продуктовые правки** из `ROADMAP_AND_ISSUES.md` (коллизия, FPS, скейл эффектов) — по возможности **после 2.1**, чтобы менять уже разрезанный код.
4. **Фаза 1.1–1.2** на сервере — можно параллельно с пунктами 2–3.
5. **Фаза 2.2a → 2.2b → 2.2c** — каждый подшаг отдельно с полным ручным прогоном.
6. **Фаза 2.3–2.4**, затем **фаза 3** (ввод 3.5 — до или вместе с мобильным UI из роадмапа).
7. **Фаза 4** — когда список сообщений и полей стабилизировался.
8. **Фаза 5** — **5.1–5.2 желательно начать до или сразу после первых крупных выносов** (коллизии, сеть): тогда регрессии ловятся автоматически; 5.4–5.5 — по необходимости.

**Тесты инкрементально:** не откладывать всё на конец фазы 5; добавлять пакетами к новым модулям (см. раздел «Юнит-тесты и рефакторинг»).

## Риски и как их сдерживать

| Риск                           | Мера                                                                                                    |
| ------------------------------ | ------------------------------------------------------------------------------------------------------- |
| Один огромный PR               | Инкременты: 2.1 без 2.2; 2.2a/b/c; после каждого — сценарий «лобби → бой → смерть → рестарт».           |
| Расхождение клиента и сервера  | Изменение протокола — один коммит/релиз на оба конца; при появлении `shared/` — одна правка констант.   |
| Дубли констант                 | Карта/протокол — в `shared/`; баланс только клиента или только сервера — синхронизировать вручную в PR. |
| Переусложнение контекста (1.3) | Лимит полей в `ctx` или отказ от 1.3 в пользу простых импортов `broadcast`/`lobbies`.                   |
| Бесконечная оптимизация        | В 3.4 зафиксировать **одну** числовую или процедурную цель и способ измерения.                          |
| Гонка за % coverage            | Не ставить цель «покрыть всю игру»; фокус на чистых функциях и критичной геометрии после выноса.        |
